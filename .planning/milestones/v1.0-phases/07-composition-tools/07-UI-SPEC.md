**HTML:**
```
<div
  class="animation-panel__drag-handle"

  aria-label="Resize animation panel"
  aria-valuenow={currentHeightPx}
  aria-valuemin={40}
  aria-valuemax={50% viewport}

```

**Styling:**
- `.animation-panel__drag-handle`: `width: 100%; height: 8px; background: var(--color-bg-secondary); border-top: 1px solid var(--color-border-tertiary); cursor: ns-resize; flex-shrink: 0; user-select: none`
- Hover state: `border-top-color: var(--color-border-secondary)` (subtle brightening)
- No accent color on the handle — it is a structural chrome element only

**Interaction:**
- Mouse down on handle → drag vertically to resize. Drag up = expand, drag down = shrink.
- Panel height clamped: min 40px, max 50% of viewport height.
- At 40px, only the handle strip is visible (panel body has `overflow: hidden`).
- Double-click on handle: toggle between 40px (collapsed) and 180px (default). No animation on toggle — instant height change.
- Panel height persisted in component state (local React useState — not in any store).

- Background fill: `var(--color-bg-primary)` drawn at start of each frame
- Polyline stroke: `var(--color-accent)` (`#6366f1`), line width 1.5px
- Unselected control points: filled circle, diameter 10px, fill `rgba(255,255,255,0.55)`, no stroke
- Hovered control point: fill `var(--color-accent)`, diameter 12px (2px larger for affordance)
- X-axis baseline: dashed line at y = midpoint, `rgba(255,255,255,0.10)`, dash pattern `[4, 4]`

**Lane canvas interaction:**
- **Click on empty area of polyline** (within 8px snap distance): insert new control point at that beat/value
- **Click on control point**: select it (highlighted fill); selected point can then be dragged
- **Drag control point horizontally**: changes `beat` position; clamped to `[0, curve.duration]`
- **Drag control point vertically**: changes `value`; clamped to property range (size/saturation/lightness: 0–100; hue: 0–360)
- **Double-click on control point**: remove it (only allowed when the curve has more than 2 points; if 2 points remain, double-click is ignored and no feedback shown)
- Cursor: `crosshair` at rest; `grabbing` while dragging a point

## States and Interactions

### Animation Panel Visibility States

| State | Condition | Panel Height | Body Visible |
|-------|-----------|--------------|--------------|
| Collapsed | `panelHeight === 40` | 40px | No (only handle strip at 40px; header is hidden) |
| Default (open) | Initial load | 188px | Yes |
| Expanded | User dragged up | Up to 50% vh | Yes |
| Re-opened via "Animate" | Was collapsed | Restores to 180px | Yes |

### Control Point States

| State | Fill | Diameter | Cursor |
|-------|------|----------|--------|
| Normal | `rgba(255,255,255,0.55)` | 10px | crosshair (from canvas) |
| Hovered | `var(--color-accent)` | 12px | grab |
| Dragging | `var(--color-accent)` | 12px | grabbing |

### Duration Input Bounds

| Condition | Behavior |
|-----------|----------|
| Value < 0.25 | Snap to 0.25 on blur |
| Value > 64 | Snap to 64 on blur |
| Non-numeric entered | Restore previous valid value on blur |
| Empty field | Restore previous valid value on blur |
| Points outside new window after duration shrink | Points are not moved; they remain at their absolute beat positions. Points with `beat > duration` are clipped at the right edge of the canvas (not rendered). They are not deleted — if duration is expanded again, they reappear. |

### Minimum Control Points Rule

- A curve always has at least 2 points. Double-clicking a control point when the curve has exactly 2 points: no-op (point is not removed, no visual feedback).
- The "+ Add Curve" flow always creates 2 initial points — this invariant is maintained at the data layer.

## Known Pitfalls

1. **canvas-area min-height** — `.canvas-area` is `flex: 1; overflow: hidden`. With a new fixed-height `.animation-panel-host` sibling below it, the flex parent (`.app-shell`) distributes height correctly only if `.canvas-area` has `min-height: 0` (browsers apply an implicit `min-height: auto` that prevents shrinking). Confirm `min-height: 0` is set or that `overflow: hidden` already implies it for the existing CanvasContainer.

2. **Lane canvas sizing** — Each lane canvas must use a ResizeObserver (same pattern as CanvasContainer) to set `canvas.width` and `canvas.height` to the container's actual pixel dimensions. Do NOT set width/height as CSS values alone — the canvas element needs integer pixel dimensions for crisp rendering. DPR scaling (devicePixelRatio) is optional at PoC scope (consistent with Pitfall 2 from Phase 4: 32px mini canvas skipped DPR).

3. **Drag handle pointer capture** — Use `setPointerCapture` / `releasePointerCapture` on the drag handle's `pointerdown` event so fast mouse movements outside the handle don't lose the drag. Without pointer capture, users dragging quickly lose the resize handle.

4. **animRate removal cross-concern** — `CellPanel.tsx` currently imports `computeLfoHz` and `BeatFraction` from `playbackStore.ts`. Removing the `animRate` section must also remove those imports to avoid TypeScript `noUnusedLocals` errors.

5. **Duration input spinner arrows** — `<input type="number">` renders native spin arrows. Hide with `-webkit-appearance: none; -moz-appearance: textfield` on `.anim-lane__duration-input`, consistent with `.bpm-widget__input` (same treatment applied in Phase 5).

6. **`animationStore.clearShape` must fire on shape removal** — When `shapeStore.removeShape` executes, `animationStore.clearShape(shapeId)` must be called to prevent orphan curve data accumulating in the store. This is a data integrity concern, not a visual one, but a missed call produces subtle state bugs.
